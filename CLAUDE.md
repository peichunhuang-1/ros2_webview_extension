# ROS2 Webview Extension

## MCP tool usage (`ros2-interfaces` server, defined in `src/mcpServer.ts`)

- If the user refers to "this topic", "this message", "the current type", "what I selected/picked", or otherwise
  relies on prior selection state instead of naming a package/topic explicitly, call `get_ros2_focus` first —
  don't say you can't tell or ask them to repeat themselves without checking it. This applies to any phrasing of
  the same idea, not just an exact match to these examples.
- `list_ros2_interfaces` lists installed `.msg`/`.srv`/`.action` *definitions*. `list_ros2_graph` lists
  topics/services/actions *currently running*. These are not interchangeable — a type can be installed without
  anything running that uses it, and a running topic's exact name only comes from the graph, never from
  `list_ros2_interfaces`.
- Resolve every ROS2 package name, interface name, and type string through these tools before using it — never
  guess or recall one from a prior turn without re-checking, since the workspace/graph can change between calls.

## Generating a GUI from a layout (`.ros2ui.json`)

Never hand-write a full-page HTML GUI directly against a `.ros2ui.json` layout — panel geometry
transcribed by hand drifts from the spec. Instead:

1. Call the `generate_gui_scaffold` MCP tool (or run **"ROS2 Webview: Generate GUI Scaffold"** from
   the Command Palette / the title-bar button on the layout editor) on the `<name>.ros2ui.json`
   file. This deterministically produces `<name>.html` (fixed panel position/size/stacking order,
   never touched by hand) and `<name>.panels/<panelId>/{panel.html,style.css,script.js}` — one
   folder per panel, only created once. Safe to re-run any time geometry or panels change; it
   never overwrites existing panel content. If `<name>.html` already exists and wasn't generated by
   this tool, the MCP call fails with an error asking you to run the command in VS Code instead, so
   the user can confirm the overwrite via the modal prompt.
2. Edit only the files under `<name>.panels/<panelId>/`:
   - `panel.html` — content fragment for that panel (no `<html>`/`<body>`; geometry is already handled).
     Tailwind utility classes are available here (`generate_gui_scaffold` links
     `<name>.panels/tailwind.min.css` into every panel's Shadow DOM automatically) — prefer them over
     hand-rolled CSS for spacing/color/typography/flex/grid layout. It's a curated safelist, not the
     full framework (see `<name>.panels/tailwind-build/README.md` for what's included and how to add
     more); if a class you need isn't styled, fall back to `style.css` for that one rule rather than
     guessing at a different Tailwind class name.
   - `style.css` — scoped by Shadow DOM to this panel only; no class-name prefixing needed, and
     rules here can never leak to or be affected by another panel. Use it for anything Tailwind's
     safelist doesn't cover, not as the default — most panels should mostly be Tailwind classes.
     Keep any "not loaded yet" / empty / placeholder state a light neutral background (e.g.
     `bg-slate-100`), never near-black — a panel legitimately waiting for its first message (an
     image viewer before the first frame, a chart before the first sample) can easily be a large
     fraction of a small canvas, and a dark placeholder there reads as broken, not empty.

     A shared `--rw-*` design-token stylesheet (`<name>.panels/design-tokens.css`) is linked once
     in `<name>.html`'s `<head>` and its custom properties are available in every panel's own
     `style.css` (CSS custom properties inherit through Shadow DOM even though class rules don't).
     Reuse `var(--rw-accent)` for every panel's primary/interactive element instead of each panel
     picking its own color — that's what keeps a multi-panel dashboard looking like one system
     instead of several unrelated widgets. Use `var(--rw-success)`/`var(--rw-warning)`/
     `var(--rw-danger)` for status colors, and `var(--rw-panel-bg)`/`var(--rw-text-muted)` for
     surfaces/placeholder text — all theme-aware (light/dark via `prefers-color-scheme`).

     For genuine visual-design requests beyond following these defaults — "make this look
     good," "redesign panel X," picking a cohesive look for a whole dashboard from scratch — use
     the `web-design-engineer` skill (`.claude/skills/web-design-engineer`, installed by "Set
     Up"). It has a real design-token declaration workflow, an anti-"AI-slop" cliché list, and
     named style recipes; skip its parts that don't fit here (brand-asset gathering, slide decks,
     React/Babel patterns) and focus on the CSS best-practices and design-principles sections.
   - `script.js` — define `window['mountPanel_<panelId>'] = function(shadowRoot, ros, bindings) {...}`;
     everything must be scoped inside that function body, never assigned to `window.*` or queried via
     bare `document.*` (Shadow DOM isolates CSS/DOM, not the JS global scope)
3. Never edit `<name>.html` directly — it's a build artifact, fully regenerated from the
   `.ros2ui.json` and the current `panel.html` contents every time the scaffold command runs.
4. Preview with `open_ros2_gui_preview` as before, pointed at `<name>.html`. You do not need to
   call `generate_gui_scaffold` again before this — opening/refreshing the preview auto-regenerates
   `<name>.html` from the layout and the current panel files first, so it always reflects your
   latest edits even if you forget step 1 on a later change.

## Node-graph architecture files (`.ros2graph.json`)

A `.ros2graph.json` file is a **ROS2 computation-graph design**, edited visually in the
"Architecture Graph" editor (open the file in VS Code, or run **"ROS2 Webview: New Architecture
Graph File"**). It is the backend companion to a `.ros2ui.json` UI layout: instead of GUI panels,
it describes the nodes and their wiring so you can scaffold the actual ROS2 packages from it.

The schema (see `src/graphTypes.ts`) has three flat arrays:

- `nodes` — rectangles, each with a `kind` (`node` = a plain ROS2 node, `controller` = a
  ros2_control controller, `hardware` = a hardware component/interface plugin), plus `name`,
  `namespace`, `language` (`cpp`/`py`/`rust`), `notes`.
- `channels` — ellipses, each with a `kind` and, for the ROS2 interface kinds, the single interface
  `type` string that is its contract (e.g. `geometry_msgs/msg/Twist`). Kinds: `topic`, `service`,
  `action` (type is a real `.msg`/`.srv`/`.action`), plus `control` and `hardware_interface` from
  ros2_control (here `type` is a free-form interface name like `position`/`velocity`, not a msg type).
- `links` — connect one node to one channel with a `role`: `publisher`/`subscriber` (topic),
  `service_client`/`service_server` (service), `action_client`/`action_server` (action),
  `control_writer`/`control_reader` (control), `interface_exporter`/`interface_consumer`
  (hardware_interface). A topic being many-to-many is just several links pointing at the same channel.

When asked to **implement / fulfill / scaffold** a `.ros2graph.json`:

- Treat the file as the source of truth for the architecture. For each node, gather its links to
  learn exactly which topics it publishes/subscribes and which services/actions it serves/calls.
- Resolve every `channel.type` through the MCP schema tools (`get_ros2_msg_schema` /
  `get_ros2_srv_schema` / `get_ros2_action_schema`) before writing pub/sub/client/server code — never
  guess message fields. Split a type string on `/` (package / kind / name) to get the args.
- Generate one ROS2 package per node in the node's chosen `language`, wiring up the publishers,
  subscribers, service servers/clients, and action servers/clients its links imply. Use each node's
  `notes` as the behavioral spec for what the implementation should actually do.
- There is deliberately **no deterministic code generator** for this file yet — you write the package
  code directly. Don't invent a scaffold command that doesn't exist; read the graph and implement it.
